# Plan 01: fix(api) — stopWhen + MCP close 이동 + consumeSseStream/onError

<objective>
Server-side root cause fix. Eliminate the three defects in `frontend/src/app/api/chat/route.ts` that produce empty-text assistant turns in production:

1. **CHAT-02:** missing `stopWhen` → default `stepCountIs(1)` stops generation after the first tool call, before the model emits text.
2. **CHAT-03:** `mcpClient.close()` in the `try/finally` races the lazy `streamText` response, cutting off in-flight tool results.
3. **CHAT-04:** no `consumeSseStream` (abort backpressure) and no `onError` (error masking) on `toUIMessageStreamResponse()`.

After this plan, a law question in production should return a tool-calls step + a text step, with `finishReason: "stop"`, not `"tool-calls"`.

**Scope guard:** ONLY `route.ts` is touched. System prompt, MCP connection fallback (try/catch around `createMCPClient`), `maxDuration`, and the 503 short-circuit are NOT modified (all in STRE Phase 2 scope per CONTEXT.md D-14).
</objective>

<must_haves>
- [ ] `streamText` call passes `stopWhen: stepCountIs(8)`
- [ ] `stepCountIs` is imported from `"ai"` alongside `streamText` and `convertToModelMessages`
- [ ] `mcpClient.close()` call is moved OUT of the `try/finally` block
- [ ] `mcpClient.close()` is called from `streamText`'s `onFinish` callback AND from its `onError` callback
- [ ] Both `onFinish` and `onError` wrap `mcpClient.close()` in `try/catch` and log rejection via `console.error`
- [ ] `toUIMessageStreamResponse({ consumeSseStream, onError })` options object is passed
- [ ] `consumeSseStream` callback drains the tee'd `ReadableStream<string>` (loop until `done`)
- [ ] `onError` on the response returns a string (does not throw); redacts any `oc=` substring to prevent `LAW_API_KEY` leakage
- [ ] `onFinish` on `streamText` also logs `finishReason` so server logs prove success criterion #8
- [ ] `next build` succeeds in the `frontend/` directory
- [ ] `npx tsc --noEmit` succeeds in the `frontend/` directory
</must_haves>

With:

```ts
  // Safe MCP close helper — callable from onFinish/onError without
  // leaking unhandled promise rejections into the serverless worker.
  const closeMcp = async () => {
    if (!mcpClient) return;
    try {
      await mcpClient.close();
    } catch (closeErr) {
      console.error("mcpClient.close() failed:", closeErr);
    }
  };

  const result = streamText({
    model: google(selectedModel),
    system: SYSTEM_PROMPT,
    messages,
    stopWhen: stepCountIs(8),
    ...(Object.keys(tools).length > 0 ? { tools } : {}),
    onFinish: async ({ finishReason }) => {
      console.log("[route.ts] streamText finishReason:", finishReason);
      await closeMcp();
    },
    onError: async ({ error }) => {
      console.error("[route.ts] streamText error:", error);
      await closeMcp();
    },
  });

  return result.toUIMessageStreamResponse({
    consumeSseStream: async ({ stream }) => {
      // Drain the tee'd copy so abort/disconnect doesn't deadlock the response.
      const reader = stream.getReader();
      try {
        while (true) {
          const { done } = await reader.read();
          if (done) break;
        }
      } finally {
        reader.releaseLock();
      }
    },
    onError: (error) => {
      console.error("[route.ts] toUIMessageStreamResponse error:", error);
      let msg: string;
      if (error instanceof Error) msg = error.message;
      else if (typeof error === "string") msg = error;
      else msg = "알 수 없는 오류가 발생했습니다.";
      // Redact LAW_API_KEY query param if it leaked into the error message.
      return msg.replace(/oc=[^&\s"]+/g, "oc=REDACTED");
    },
  });
}
```

**Removed:** the outer `try { ... } finally { if (mcpClient) await mcpClient.close(); }` wrapper. The MCP close is now in the streamText callbacks (CHAT-03). The function body ends with `return result.toUIMessageStreamResponse({...});`.
</action>

## Security threats introduced by this plan

| Severity | Threat | Mitigation | Verification |
|----------|--------|------------|--------------|
| low | `mcpClient.close()` rejection in `onFinish`/`onError` becomes an unhandled promise rejection, potentially crashing the serverless worker. | `closeMcp` helper wraps `close()` in `try/catch` and logs via `console.error`. Never re-throws. | Code review: grep for `try { await mcpClient.close()` inside `closeMcp`. |
| low | `onError` on `toUIMessageStreamResponse` exposes raw error messages to the client. If an error references the MCP URL (`https://...?oc=API_KEY`), `LAW_API_KEY` would leak. | `onError` runs `msg.replace(/oc=[^&\s"]+/g, "oc=REDACTED")` before returning. | Acceptance: `grep "oc=REDACTED" frontend/src/app/api/chat/route.ts`. |
| low | `consumeSseStream` not draining the tee'd copy could cause backpressure and block the main response stream (the very bug we're trying to avoid). | Callback loops `reader.read()` until `done: true`. | Code review: grep for `while (true)` and `{ done } = await reader.read()` inside `consumeSseStream`. |

Highest severity: **low**. Per `workflow.security_block_on = "high"`, this plan is not blocked.

No new auth, PII, or untrusted input paths. No data persistence change. No cross-origin or CSRF surface change.
