# Feature Research — AI Streaming Chat UX (Tool-Calling)

**Domain:** Internal AI chat assistant with MCP tool calls (Korean law search)
**Researched:** 2026-04-13
**Confidence:** MEDIUM-HIGH — AI SDK 6 specifics verified against official docs (HIGH); peer-product UX patterns from web search verified against multiple sources (MEDIUM)
**Scope:** Subsequent milestone — features must fit a 1-week window on top of an existing Next.js 16 + AI SDK 6 codebase

---

## Context Constraints (read first)

This research is opinionated for the Korean Law Bot's actual situation:

- **Single internal user right now** (the developer); rolling out to the company at end of sprint
- **Vercel free tier, hard 60s function timeout** — limits how long a single tool-calling loop can run
- **localStorage-only persistence** (no server DB this sprint) — anything that depends on server state is out
- **Frozen stack:** Next.js 16, React 19, AI SDK 6.0.158, `useChat`, MCP via `glluga-law-mcp.fly.dev`
- **The blocker:** AI answers currently render as empty cards because `getMessageText()` only handles `parts` with a literal `"text"` field, ignoring `tool-call` / `tool-result` / typed tool parts. Anything that touches part rendering is on the critical path.
- **Existing surface:** chat container, message bubbles with markdown, copy/favorite/export, model selector, example prompts, login. No conversation sidebar UI in `chat-container.tsx` (that lives in `page.tsx` — out of scope to redesign).

Every feature below is graded against "can a solo dev ship this in a week without touching the frozen stack."

## Sub-Question 1 — In-Progress Tool-Call Visibility

### What peer products actually do (late 2025 / early 2026)

| Product | Pattern | Source |
|---|---|---|
| **Claude.ai** | Inline status that flips verb tense — "Reading docs.example.com" → "Read docs.example.com" once complete. Tool-search notifications surfaced as brief notifications, not inline message clutter. Tool calls are collapsible by default. | Anthropic engineering blog, BrightCoding 2026 Claude guide |
| **ChatGPT (GPT-5/5.2)** | Collapsible "Thinking" / reasoning blocks with elapsed time counter. "Auto / Fast / Thinking" mode chips in composer. Streaming token-by-token with markdown rendering during stream. | OpenAI release notes, IntuitionLabs 2025 comparison |
| **Perplexity (Sonar Pro / Pro Search)** | Multi-step status list: "Searching the web", "Reading sources", "Synthesizing answer". Each step is a card that expands to show what was searched. Real-time "thought streaming" toggle shows reasoning live. Sources sidebar appears as soon as first source is found. | Perplexity changelog 2025-2026, Perplexity help center |
| **Vercel v0** | "Tool invocations are visible and interactive" — every tool call rendered as its own card in the message stream with input + output sections. Multi-step UI is first-class. | Vercel Academy: Tool Use, Multi-Step & Generative UI |
| **Cursor chat** | Collapsible tool blocks ("Read file", "Searched codebase") inline in the assistant message, with file name / query string visible without expanding. Click to expand for full input/output. | Vercel Academy: Multi-Step UI; Cursor docs |

**Cross-product convergence (this is the table-stakes pattern):**

1. **Verb-tense status** — present continuous while running ("Searching 근로기준법..."), past tense + checkmark when done ("Searched 근로기준법")
2. **Visible argument preview** — the actual query / file / URL appears next to the verb. Not just a spinner.
3. **Collapsed by default, expandable on click** — full args + raw output hidden but accessible
4. **Inline in the assistant message** — not a separate panel, not a toast
5. **One block per tool call** — multi-step calls stack vertically as a checklist

### What AI SDK 6 actually gives you (HIGH confidence — verified against `ai-sdk.dev`)

The `UIMessage.parts` array contains typed tool parts of shape `tool-{toolName}` (e.g. `tool-search_law`). Each tool part carries a `state` field with four values:

| State | Meaning | What to render |
|---|---|---|
| `input-streaming` | Tool args being generated by the model token-by-token | Shimmer + tool name, partial arg preview if available |
| `input-available` | Args complete, tool about to run / running | "Searching {input.query}..." with spinner |
| `output-available` | Tool returned successfully | "Searched {input.query}" with checkmark, expandable result |
| `output-error` | Tool threw | "Search failed: {errorText}" with retry hint |

**This is the load-bearing fact for both bug fix #1 and feature #3.** The current `getMessageText()` only pulls `text` parts — it silently drops every tool part and every text part that comes after a tool part (which is why answers render empty: Gemini emits `tool-call → tool-result → text`, and the consumer needs to walk all three).

### Features

#### Table Stakes

| Feature | Why expected | Complexity | 1-week feasible? |
|---|---|---|---|
| **Render text parts in order with tool parts interleaved** | This is literally the bug. Without it, no answers visible. | LOW | YES — must |
| **Verb-tense status chip per tool call** ("법령 검색 중: 근로기준법" → "검색 완료: 근로기준법") | Every peer product does this. Internal users will lose trust in 1 minute without it because they currently only see "검색 중...". | LOW-MED | YES |
| **Show the actual query/argument next to the verb** | "검색 중..." with no context = users assume it's stuck. Showing "검색 중: 근로기준법 제60조" makes 60s feel like progress. | LOW | YES |
| **Collapsible tool block** (collapsed by default, click to see full input + raw JSON output) | Standard across Claude / Cursor / v0. Lets power users debug without noise for normal use. | LOW | YES — `<details>` element is one line |
| **Multiple tool calls stack as a vertical checklist** | Gemini routinely does `search_law` → `get_law_text`. Users need to see both steps, not just the last one. | LOW | YES — falls out of part-by-part rendering |
| **Visual distinction between tool block and answer text** | Otherwise the tool noise blends into the answer. Bordered card / muted background is enough. | LOW | YES |

#### Differentiators

| Feature | Value | Complexity | 1-week feasible? |
|---|---|---|---|
| **Korean-localized tool names** ("법령 검색" not "search_law", "판례 검색" not "search_decisions") | Internal users don't read English. Map MCP tool name → Korean label in a small dict. | LOW | YES |
| **Show MCP tool result count** ("법령 검색: 근로기준법 → 12건 발견") | Lets users gauge "did the search actually find anything" before reading the synthesis. Pulled from `tool-result` payload. | LOW-MED | YES if the MCP response shape is stable |
| **Live arg preview during `input-streaming`** | Shows the model "deciding" what to search. Feels alive. | MED | YES — but only worth it if `input-streaming` deltas actually arrive (verify with the real MCP server, not all providers stream tool args) |
| **Per-tool icons** (📖 for `get_law_text`, 🔍 for `search_law`, ⚖️ for decisions) | Quick scan of "what kind of work is the bot doing." | LOW | YES |
| **Elapsed time on running tool calls** ("검색 중... 12s") | Steals the pattern from ChatGPT thinking blocks; helps users know whether to wait or refresh. | LOW-MED | YES |

#### Anti-Features (do NOT build this sprint)

| Anti-feature | Why it looks tempting | Why skip | Alternative |
|---|---|---|---|
| **Custom "thinking" / reasoning panel** like ChatGPT's | Looks impressive | Gemini 2.5 Flash via AI SDK doesn't expose discrete reasoning tokens in a way that matches ChatGPT's UI; faking it is theater | Use the verb-tense status chip pattern instead |

| **Editable tool calls** (let user modify args before re-running) | Power-user fantasy | Requires a "rerun this tool" round-trip not supported by `useChat` out of the box; massive scope creep | Out-of-scope; user re-asks the question if needed |

## Sub-Question 2 — Streaming Error / Interrupt UX

### What peer products actually do

- **ChatGPT** — Inline error banner inside the assistant message bubble ("Error in message stream"), with a **Regenerate response** button that re-sends the last user message. Partial output is **discarded** from view but the failed assistant turn stays in the transcript as a visible failure. (Source: CometAPI, hostingseekers fix guides)
- **Claude.ai** — Inline error pill inside the running message; partial content **stays visible** above the error pill, with a "Try again" button that re-runs from the same user turn. The failed turn is replaced rather than appended. (Source: Anthropic + community reports)
- **Perplexity** — Inline retry; if a search step fails mid-stream, the failed step shows a red icon while later steps continue. Partial answer preserved.
- **AI SDK 6 useChat** — exposes `error` object, `regenerate()` function, and `status === 'error'`. Recommends generic error messages, not exposing server details. Patterns split between (a) keep partial message + show inline error, or (b) `setMessages(messages.slice(0, -1))` to drop incomplete and retry clean. (Source: ai-sdk.dev/docs/ai-sdk-ui/error-handling — verified)

### What internal users specifically need (this app)

The app has three real failure modes, all already happening in production:

1. **60s Vercel timeout** — function dies mid-stream. User sees... nothing useful right now.
2. **MCP server 503 / "Max sessions"** — already detected in `route.ts:200-210` and returned as a 503 with Korean message
3. **MCP `createMCPClient` connection failure** — currently logs to console and silently degrades to "no tools," producing useless answers

### Features

#### Table Stakes

| Feature | Why expected | Complexity | 1-week feasible? |
|---|---|---|---|
| **Inline error banner inside the assistant message bubble** (not a toast) | Already partially implemented in `chat-container.tsx:202-215` but lives outside the message stream. Move it inline so it's anchored to the failed turn. | LOW | YES |
| **Distinct, human-readable Korean messages for the three failure modes** (timeout / 503 / MCP-down) | Generic "오류 발생" makes internal users assume the product is broken. The current 503 detection is correct in spirit but needs to extend to timeout + MCP-down. | LOW | YES |
| **Retry button on the failed turn** that calls `regenerate()` | Single biggest UX win for free; one button wired to `useChat`'s `regenerate`. | LOW | YES |
| **Preserve partial answer text + tool-call history above the error** | Users want to see how far the bot got. Don't `slice(0,-1)` — keep the partial message and append the error block under it. | LOW-MED | YES |
| **Stop button** while streaming (calls `useChat`'s `stop()`) | Standard across all peer products. Lets users abort a clearly-wrong path without waiting 60s. | LOW | YES — `useChat` already exposes it |
| **Distinguish "MCP unreachable" from "model error"** at the API route level so the UI can show a banner like "법령 검색 도구에 연결할 수 없습니다 — 답변이 일반 지식 기반일 수 있습니다" | Current code silently degrades. Internal lawyers must know when the bot is bullshitting from training data vs answering from law.go.kr. | MED | YES — touch `route.ts` to set a flag in stream metadata |

#### Differentiators

| Feature | Value | Complexity | 1-week feasible? |
|---|---|---|---|
| **Auto-retry once on transient 503** with exponential backoff before showing error to user | Hides flapping MCP server. Cheap. | LOW-MED | YES — guard with single-retry max so we don't double-bill function-seconds |
| **Timeout countdown** — show "응답 대기 중... (45s / 60s)" in last 15s of the timeout window | Sets expectations, prepares user for the "timeout" error if it comes | MED | MAYBE — only if there's slack at end of week |
| **"Resume where it failed"** by sending the partial answer + last user turn back to the model | Magical when it works. | HIGH | NO — defer to v2; depends on prompt engineering and model behavior |

#### Anti-Features

| Anti-feature | Why tempting | Why skip | Alternative |
|---|---|---|---|
| **Toast popup for errors** | Flashy, "modern" | Disconnected from the failed turn; user has to scroll back to find which message failed | Inline in the bubble |
| **Auto-clear failed messages after N seconds** | "Cleaner" UI | Destroys debugging info; loses failed assistant turn the user might want to screenshot for the dev | Keep it visible until user dismisses or retries |
| **Full Sentry / monitoring integration this sprint** | "We need observability" | Explicitly out of scope per PROJECT.md. `console.error` is fine for v1. | Defer to v2 |
| **Streaming over WebSockets / SSE-via-Edge-runtime to dodge 60s** | Solves the timeout | Major architecture change, requires moving off Vercel free tier or rewriting to Edge runtime; outside the frozen stack | Accept 60s ceiling, design errors to handle it gracefully |
| **Custom error retry queue** with persistence | "Reliability" | localStorage doesn't need it for a single user; over-engineered | One retry button is enough |

- 6 example questions as cards
- 4 usage tips

### Loading states (where the actual work is)

## Feature Dependencies

```
[Bug fix #1: render text+tool parts in order]
    ├── enables ──> [Tool-call status chips]
    │                    └── enables ──> [Per-tool icons + KR labels]
    │                    └── enables ──> [Collapsible tool block]
    │                    └── enables ──> [Show result count]
    ├── enables ──> [Skeleton loading bubble]
    └── enables ──> [Streaming stability error UI]
                         └── enables ──> [Inline retry button]
                         └── enables ──> [Distinct timeout / 503 / MCP-down banners]

[Backward-compat for stored Message.content]
    └── enables ──> [All conversation persistence work]

[localStorage atomic save (already gated)]
    └── enables ──> [Conversation sidebar continuing to work]

[Chainlit deletion]
    └── independent of UX work; pure cleanup
```

### Dependency notes

- **Bug fix #1 is the keystone.** Every other UX feature in this sprint either depends on it or is meaningless without it. If the part-rendering refactor isn't done first, tool-call chips have nothing to render against.
- **Tool-call status UX and streaming-error UX share the same code path** (`UIMessage.parts` walking + `useChat` status handling). Build them in the same PR.
- **Conversation backward-compat is a silent killer.** Easy to forget; will look like a regression bug ("all my old conversations are blank") when you ship.
- **Chainlit deletion is independent** and can run in parallel with anything; do it last so you don't accidentally break something the legacy code still touched.
